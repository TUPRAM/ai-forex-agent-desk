Architecture Overview

The AI Forex Agent Desk is a multi-agent trading system built with CrewAI. It consists of 11 specialized agents organized into two operating loops, coordinated by a central Orchestrator.

The system embodies one core principle: edge must be earned before it can be operated. The Research Loop earns edge through rigorous validation. The Trading Loop deploys only proven, approved strategies within defined risk limits.

No agent acts autonomously. Every significant action flows through the Orchestrator, which acts as the system's central nervous system — coordinating, approving, and logging all inter-agent communications.

The system is deliberately not a YOLO trading bot. It is a disciplined, auditable, reproducible system designed to survive the market long-term rather than optimize for short-term returns.

The Two Loops

Loop A — Research Loop is an offline, non-real-time process. Its job is to generate and validate trading strategies. It runs continuously in the background, processing hypotheses, running backtests, and passing validated strategies to the Trading Loop.

The Research Loop involves: the Data Engineer (clean data), Alpha Research (idea generation), Backtest & Validation (proof of edge), Risk Manager (position sizing and envelope), and Governance (sign-off and version control).

Loop B — Trading Loop is the live, real-time operational layer. It takes only pre-approved, governance-signed strategies and executes them within the defined risk envelope. No strategy enters this loop without a version hash and Governance sign-off.

The Trading Loop involves: the Orchestrator (coordination), Regime Agent (market classification), Execution Agent (broker orders), Trade Management (position management), Monitoring (real-time surveillance), and Reporting (end-of-day summary).

Safety Gates

The system has four primary safety gates that must all be satisfied before a strategy can trade live.

Gate 1 — Backtest Validation: Walk-forward test must pass on unseen data. In-sample performance is necessary but not sufficient. Out-of-sample survival is required.

Gate 2 — Risk Envelope: The Risk Manager must define an approved lot size, stop-loss method, maximum hold time, and daily exposure limit for every strategy before it can trade.

Gate 3 — Governance Sign-off: The Governance Agent must issue a signed release with version hash, change summary, backtest results, and a rollback plan. No exceptions.

Gate 4 — Regime Clearance: Before each trade session, the Regime Agent must issue a TRADE_ALLOWED flag. If conditions are hostile (high spread, news risk, adverse regime), no trades execute that session.

All four gates must be cleared. A strategy that passes three of four does not trade. There are no partial approvals.

Build Order

The recommended build order reflects a dependency hierarchy. Build the foundation before adding intelligence. Never add a higher-level agent before the agents it depends on are tested and working.

Step 1 — Data Engineer: Nothing else works without clean, versioned data. Build this first and stress-test it aggressively.

Step 2 — Backtest Agent: Build the judge before building the ideas. This ensures honest evaluation from the start.

Step 3 — Risk Manager: Define risk parameters before you ever consider execution. Risk is not an afterthought.

Step 4 — Execution Agent: Paper-trade and test thoroughly before going live. Handle requotes, broker rejections, and partial fills.

Step 5 — Monitoring: Set up Telegram alerts before going live with a single cent. Silent failures are more dangerous than noisy ones.

Step 6+ — All Others: Alpha Research, Regime Agent, Trade Management, Governance, and Reporting can be added incrementally once the foundation is solid.

Agent Dictionary

A quick reference for all 11 agents and their primary function in the system.

Orchestrator
Central coordinator. Routes information between all agents. Issues daily trading plans, approves execution orders, and triggers emergency protocols.
Data Engineer
Builds and maintains the clean, versioned gold dataset. Handles ingestion, quality checks, normalization, and gap detection.
Regime Agent
Classifies current market conditions (trend, chop, volatile, news risk). Issues TRADE_ALLOWED or STAND_DOWN flags before each session.
Alpha Research
Generates trading hypotheses from regime data and pattern analysis. Feeds the backtest pipeline.
Backtest & Validation
Tests strategies on historical data. Applies walk-forward testing, stress testing, and parameter sensitivity. Intentionally harsh.
Risk Manager
Calculates position sizes, enforces loss limits, defines risk envelopes, and triggers kill-switch when limits are breached.
Execution Agent
Places orders via MT5 Python API. Handles spread checks, requotes, partial fills, retry logic, and fill confirmation.
Trade Management
Manages open positions from entry to exit. Applies trailing stops, time stops, partial profits, and ATR-dynamic adjustments.
Monitoring Agent
Real-time system surveillance. Watches data feeds, PnL deviation, spread, and broker connection. Can halt trading unilaterally.
Governance Agent
Change management and audit trail. Signs off on all strategy deployments with version hash, test results, and rollback plan.
Reporting Agent
Produces weekly PM-style reports summarizing performance, regime context, risk events, and research progress.

Glossary

Walk-Forward
Out-of-sample validation method. Train on period A, test on period B, verify the strategy didn't just memorize noise.
Kill-Switch
Emergency halt mechanism. Triggered by the Monitoring Agent or Risk Manager when hard limits are breached. Closes or protects all positions.
Risk Envelope
The set of approved parameters for a live strategy: lot size, stop-loss method, max hold time, and daily exposure cap.
Regime
The current market state: trending, range-bound, high-volatility, or news-driven. Different strategies work in different regimes.
Version Hash
A unique identifier for each deployed strategy version. Ensures reproducibility — same hash should produce same results.
Expectancy
Expected profit per trade, in multiples of the risk (R). A positive expectancy over many trades is the definition of edge.
Gold Dataset
The master clean data file used by all agents. Versioned, timestamped, quality-checked. Single source of truth for all analysis.
ATR
Average True Range. Used to normalize stop-loss sizes, position sizes, and volatility-regime detection across instruments.

FAQ

Why does the system kill 90–99% of strategy ideas?
Because most "edges" are statistical illusions. The backtest and walk-forward process is designed to be harder than the market itself. A strategy that survives this pipeline has earned the right to trade real capital.
Can any agent override the Risk Manager?
No. Risk Manager decisions are final and cannot be overridden by Alpha Research, Execution Agent, or even the Orchestrator for hard limit enforcement. Signal strength is irrelevant when the risk budget is exhausted.
What happens after a kill-switch event?
The system enters a locked state. Trading is suspended. Governance creates a mandatory post-mortem incident report. The human operator must review the report and sign off on a restart before any new trades can execute.
Why does Build Order matter?
Dependencies. The Execution Agent depends on the Risk Manager for lot sizes. The Risk Manager depends on the Data Engineer for current ATR. The Governance Agent depends on the Backtest Agent for validation results. Build in dependency order to avoid untestable components.
How often does the Research Loop run?
Continuously, but with different cadences per stage. Hypothesis generation runs daily on new regime data. Quick screens run hourly. Full backtests run overnight. Walk-forward tests are queued and run during off-market hours to avoid compute contention with live trading.

Technical Notes & Assumptions

Broker: MetaTrader 5 (MT5) with Python API. All execution uses the `mt5` Python library. Orders include stop-loss and take-profit at placement time — no naked positions.

Framework: CrewAI for agent orchestration. Each agent is defined with role, goal, backstory, tools, and allow_delegation settings. The Orchestrator has delegation enabled; all other agents do not.

Data: All analysis uses a versioned, locally-stored gold dataset. No agent fetches raw data directly — all data comes through the Data Engineer's validated pipeline. Reproducibility requires locking both the data version and random seed.

Instruments: Primary instruments are XAUUSD and USDJPY. GBPUSD is in the research pipeline. The system is designed for 2–5 instruments, not broad portfolio coverage.

Position Sizing: Fixed-fractional risk model. 1% of current equity per trade, calculated from live ATR and instrument pip value. Recalculated fresh for every execution.

This system is built for disciplined, reproducible, institutional-quality operation. The goal is not to maximize return — it is to survive and compound long-term through risk-adjusted excellence.
